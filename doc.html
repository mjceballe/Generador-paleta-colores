Sistema Algorítmico de Color

Documentación para Desarrolladores

Este documento detalla las decisiones de diseño y las soluciones en código empleadas para construir el motor dinámico de paletas de color. El sistema soporta escalas secuenciales y categóricas garantizando accesibilidad mediante tokens de diseño.

1. Arquitectura y Espacios de Color

Decisión de Diseño: Se optó por una arquitectura híbrida. El HSL se utiliza para las paletas secuenciales por su predictibilidad en luminosidad (L). El OKLCH se utiliza internamente para la rotación de matices (Hue) en paletas cualitativas, ya que es perceptualmente uniforme y evita distorsiones de brillo en diferentes matices.

2. Motor Cualitativo Inteligente (Treemaps)

Decisión de Diseño: Rotar el matiz linealmente genera progresiones sintéticas y poco naturales. Utilizamos una matriz de saltos asimétricos predefinidos para maximizar la distinción visual entre categorías adyacentes. 

// Matriz de rotación asimétrica para 8 categorías
const HUE_OFFSETS_8 = [0, 227, 57, 172, 287, 112, 330, 25];

Además, implementamos un detector de temperatura cromática en HSL. Si la marca es cálida, el algoritmo busca un punto de inicio análogo hacia el espectro frío (salto de +35°) y viceversa, manteniendo la armonía sin perder contraste. 

// Compensación análoga inteligente para evitar esquemas mecánicos
const isWarmBrand = (hIn >= 0 && hIn <= 90) || (hIn >= 300 && hIn <= 360);
let seedHueTreemap = isWarmBrand ? (seedHue + 35) : (seedHue - 35);

3. Motor Secuencial y Lógica de Datos (Dona)

Decisión de Diseño: Para garantizar la legibilidad al reducir la cantidad de datos, el sistema no utiliza los colores de forma lineal (1, 2, 3...), sino que aplica un filtrado selectivo (cherry-picking) para mantener el máximo contraste entre segmentos.

Reglas de asignación por volumen de datos:

  • 0 datos: Estado vacío (Gris neutro).
  • 1 dato: Usa exclusivamente el 4to color (Índice 3).
  • 2 datos: Usa el 4to y 5to color (Índices 3, 4).
  • 3 datos: Usa el 1ro, 3ro y 5to color (Índices 0, 2, 4).
  • 4 datos: Usa el 1ro, 4to, 5to y 6to color (Índices 0, 3, 4, 5).
  • 5 datos: Usa el 1ro, 3er, 4to, 5to y 6to color (Índices 0, 2, 3, 4, 5).
  • 6 datos: Usa la escala completa (Índices 0 al 5).

4. Accesibilidad y Tokens de Diseño

Decisión de Diseño: La legibilidad del texto sobre los bloques de color se resuelve mediante el cálculo de Luminancia Relativa (WCAG 2.1). En lugar de usar valores fijos, el sistema mapea el resultado a tokens de diseño semánticos.

El umbral crítico es 0.179. Este valor determina matemáticamente si el ojo humano requiere un tono claro u oscuro para leer el contenido.

  • Si L > 0.179: Aplicar token text/body (Contraste oscuro).
  • Si L ≤ 0.179: Aplicar token text/on-action (Contraste claro/blanco).
function getAccessibleTextColor(hex) {
    const L = calculateRelativeLuminance(hex);
    // Mapeo de lógica a Tokens de Diseño
    return L > 0.179 ? "text/body" : "text/on-action";
}
